# Резюме проекта: Цифровой двойник с отказоустойчивым управлением

## Краткое описание

Проект представляет собой **отказоустойчивую систему цифрового двойника** технологического процесса управления уровнем жидкости в резервуаре. Система построена на микросервисной архитектуре с использованием Docker контейнеров и протокола OPC UA, обеспечивая высокую надежность через автоматическое переключение между основным и резервным контроллерами.

## Ключевые особенности

### 🛡️ Отказоустойчивость
- **Горячий резерв**: резервный контроллер работает в режиме мониторинга
- **Автоматическое переключение**: при отказе основного контроллера резервный активируется через 2 секунды
- **Автоматическое восстановление**: при возврате основного контроллера управление передается обратно
- **Heartbeat мониторинг**: контроллеры отслеживают состояние друг друга через OPC UA
- **Сохранение состояния PID**: интегральная составляющая и состояние регулятора сохраняются при переключениях
- **Плавное переключение**: нет скачков управляющего воздействия при смене контроллеров

### 🏭 Технологический процесс
- **Модель бака**: высота 3м, диаметр 2м, плотность жидкости 1000 кг/м³
- **Регулирующий клапан**: расчет по ГОСТ формулам, Kvs = 800
- **ПИД-регулирование**: настроенное по ГОСТ стандартам (Kp=15.0, Ki=0.1, Kd=1.0)
- **Целевая производительность**: стабилизация уровня на уставке 1.6м

### 🐳 Микросервисная архитектура
- **5 Docker контейнеров**: OPC UA сервер, модель процесса, основной и резервный контроллеры, watchdog
- **Единый файл контроллера**: `universal_controller.py` работает в режимах primary/backup
- **Централизованная конфигурация**: все параметры в `config.json`
- **Изолированная сеть**: безопасное взаимодействие через Docker network
- **Watchdog мониторинг**: отдельный контейнер для наблюдения за состоянием контроллеров

## Компоненты системы

### 1. OPC UA Server
- **Функция**: Централизованное хранение данных
- **Переменные процесса**: PV_level, SP_level, OP_valve, flows
- **Переменные отказоустойчивости**: статусы контроллеров, heartbeats, active_controller
- **Переменные состояния PID**: сохранение интегральной составляющей, предыдущей ошибки и производной
- **Порт**: 4840

### 2. Process Model
- **Функция**: Симуляция физического процесса
- **Компоненты**: TankModel (материальный баланс), Valve (ГОСТ расчеты)
- **Обновление**: каждые 100мс
- **Независимость**: работает даже при отказе контроллеров

### 3. Universal Controller (Primary)
- **Функция**: Основное управление процессом
- **Режим**: Активное ПИД-регулирование
- **Heartbeat**: отправляет каждую секунду
- **Восстановление**: автоматически возвращает управление при возврате в строй

### 4. Universal Controller (Backup)
- **Функция**: Резервное управление
- **Режим**: Мониторинг основного контроллера
- **Активация**: при timeout основного (5сек) + задержка (2сек)
- **Возврат**: автоматически передает управление при восстановлении основного


## Сценарии работы

### Нормальная работа
1. **Основной контроллер** активен, управляет процессом
2. **Резервный контроллер** мониторит, отправляет heartbeat
3. **Модель** непрерывно рассчитывает процесс
4. **OPC UA сервер** обеспечивает обмен данными

### Отказ основного контроллера
1. **Основной контроллер** перестает отправлять heartbeat
2. **Резервный контроллер** обнаруживает timeout (5 секунд)
3. **Задержка переключения** 2 секунды для избежания ложных срабатываний
4. **Резервный контроллер** активируется и берет управление
5. **Модель** продолжает работу без прерывания

### Восстановление основного контроллера
1. **Основной контроллер** восстанавливается и отправляет heartbeat
2. **Резервный контроллер** обнаруживает восстановление
3. **Основной контроллер** устанавливает себя как активный
4. **Резервный контроллер** возвращается в режим мониторинга
5. **Управление** плавно передается основному контроллеру

## Команды управления

### Запуск системы
```bash
# Полный запуск
docker-compose up --build -d

# Проверка статуса
docker-compose ps

# Просмотр логов
docker-compose logs -f
```

### Тестирование отказоустойчивости
```bash
# Остановка основного контроллера
docker-compose stop controller-primary

# Наблюдение за активацией резервного
docker-compose logs controller-backup --tail=10

# Восстановление основного
docker-compose start controller-primary

# Проверка возврата управления
docker-compose logs controller-backup --tail=10
```

### Мониторинг системы
```bash
# Логи основного контроллера
docker-compose logs controller-primary -f

# Логи резервного контроллера  
docker-compose logs controller-backup -f

# Логи модели процесса
docker-compose logs model -f
```

## Индикаторы состояния

### В логах основного контроллера
- `ОСНОВНОЙ PID: SP=1.600м, PV=1.587м, OP=45.2%` - активное управление
- `ОСНОВНОЙ (мониторинг)` - режим мониторинга (после восстановления)

### В логах резервного контроллера
- `РЕЗЕРВНЫЙ (мониторинг): SP=1.600м, PV=1.587м, OP=45.2%` - нормальный режим
- `⚠️ Основной контроллер недоступен` - обнаружение отказа
- `🚨 ПЕРЕКЛЮЧЕНИЕ НА РЕЗЕРВНЫЙ КОНТРОЛЛЕР!` - активация резервного
- `РЕЗЕРВНЫЙ ПИ расчет: SP=1.600, PV=1.587, Output=45.2` - активное управление
- `✅ Основной контроллер восстановлен` - возврат к мониторингу

## Преимущества решения

### 🔧 Техническая надежность
- **Непрерывность управления**: система никогда не остается без контроллера
- **Быстрое переключение**: активация резервного за 7 секунд (5+2)
- **Автоматическое восстановление**: возврат к основному без вмешательства оператора
- **Изоляция отказов**: отказ одного компонента не влияет на остальные

### 🏗️ Архитектурная гибкость
- **Модульность**: каждый компонент независим и заменяем
- **Масштабируемость**: легко добавлять новые компоненты или процессы
- **Стандартизация**: использование промышленного протокола OPC UA
- **Конфигурируемость**: все параметры настраиваются через config.json

### 🚀 Эксплуатационные преимущества
- **Простота развертывания**: одна команда для запуска всей системы
- **Прозрачный мониторинг**: детальное логирование всех операций
- **Безопасность**: изолированная сеть, непривилегированные пользователи
- **Производительность**: оптимизированные алгоритмы и минимальные задержки
